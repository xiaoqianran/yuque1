## 1. 文档目标

定义复刻语雀核心子集的**总体技术架构、选型、关键机制与工程边界**，作为：

- 领域模型 / 库表设计（Design-01）的上游约束

- 编辑器 POC、短信接入、部署拓扑的验收标准

## 2. 设计原则与约束

| 原则 | 落地 |
|------|------|
| 对齐 PRD 冻结决策 | 手机号强制验证；乐观锁冲突三选一；folder/doc 同树；**doc 必须可挂子节点** |
| 模块化单体优先 | MVP 单进程 API，按领域分包；禁止过早微服务 |
| 契约优先 | OpenAPI 3 为前后端 SSOT |
| 安全默认 | 私有默认、HttpOnly 会话、短信限流、MD 防 XSS |
| 可自托管 | Docker Compose 一键；配置与密钥外置 |
| 可演进 | 预留 version 快照、审计、搜索索引扩展点 |

### 2.1 规模假设（设计容量，非 SLA 承诺）

| 项 | 假设 |
|----|------|
| 用户 | ≤ 500 / 单部署 |
| 文档 | ≤ 5 万篇 |
| 单篇 Markdown | 常规 &lt; 100KB；上限建议 2MB |
| 单库树节点 | ≤ 5,000（需可加载；前端虚拟化一期增强） |
| 并发写同一文档 | 低；靠乐观锁，非实时协同 |

### 3.1 逻辑架构

```

│                        Client (Web)                         │

                            │ HTTPS / JSON (REST)
                            │ Cookie: session (HttpOnly)
┌───────────────────────────▼─────────────────────────────────┐
│                     API (Modular Monolith)                  │
│  Gateway 中间件: Auth / RateLimit / RequestId / Audit       │
│  ┌──────────┬──────────┬──────────┬──────────┬───────────┐  │
│  │ Identity │Workspace │   KB     │ Document │  Share    │  │
│  │  SMS     │          │  Tree    │ Content  │  Media*   │  │
│  └──────────┴──────────┴──────────┴──────────┴───────────┘  │
```

| 环境 | 说明 |
|------|------|
| local | Compose + 短信 Mock |
| dev/staging | 可接真实短信或 Mock；独立 DB |

### 3.3 请求链路

1. 浏览器 → Nginx（静态 + `/api` 反代）  
2. API 中间件注入 `requestId`、鉴权、限流  
3. Application Service 编排领域逻辑与事务  
4. 写操作按需写 `audit_logs` / `content_revisions`  
5. 统一错误体返回

## 4. 技术选型（冻结建议）

### 4.1 选型表

| 层级 | 选型 | 理由 |
|------|------|------|
| 前端 | **React 18 + TypeScript + Vite** | 生态、编辑器集成成熟 |
| 路由 | React Router 6/7 | 标准 SPA |
| 服务端状态 | **TanStack Query** | 缓存、重试、与自动保存配合 |
| 客户端状态 | Zustand | 轻量：编辑器 dirty、冲突 UI 状态 |
| UI | **Ant Design 5** | 企业表单/树/弹窗齐全 |
| Markdown 编辑 | **CodeMirror 6**（编辑）+ **markdown-it / remark**（渲染） | MVP 稳、XSS 可控；见 §8 |
| Markdown 安全渲染 | **DOMPurify** + 禁用 raw HTML 或白名单 | 防 XSS 必做 |
| 后端 | **NestJS + TypeScript** | 与前端同语言；模块边界清晰；OpenAPI 友好 |
| ORM | **Prisma** | 迁移体验好；或 TypeORM（二选一，推荐 Prisma） |
| DB | **PostgreSQL 16** | 事务、树查询、JSONB 扩展位 |
| 缓存 | **Redis 7** | 会话、验证码、限流 |
| 会话 | **Redis 存储 Session + HttpOnly Cookie** | Web 优先；较纯 JWT 易吊销 |
| 短信 | 接口抽象 `SmsProvider`；生产阿里云/腾讯云；**local Mock** | PRD 强制手机验证 |
| 对象存储 | MinIO（S3 API），P1 | 图片上传 |
| API 文档 | OpenAPI 3 + `@nestjs/swagger` 生成 | 契约 SSOT |
| 包管理 / Monorepo | **pnpm workspace + Turborepo**（或仅 pnpm） | 统一 lint/build |
| 日志 | pino / nestjs-pino 结构化 JSON | requestId 贯通 |
| 测试 | Vitest（前端）/ Jest（Nest）+ Supertest；E2E Playwright（关键路径） | 质量门禁 |
| CI | GitHub Actions / GitLab CI | lint + test + build |

### 4.2 明确不选（MVP）

| 项 | 原因 |
|----|------|
| 微服务拆分 | 团队与规模不匹配 |
| GraphQL 作主 API | REST 更贴 CRUD/权限模型 |
| 仅 JWT 无服务端会话 | 吊销与强制下线成本高 |
| Elasticsearch（MVP） | 运维重；标题过滤即可 |
| 实时 WebSocket 协同 | PRD 排除 |
| 富文本双模编辑器（MVP 主路径） | 先 MD 闭环；TipTap 作一期 POC 候选 |

### 4.3 语言与版本基线

- **Node.js 22**（团队已采用；工程 `engines.node` / `.nvmrc` / CI 镜像统一锁 **22**，禁止默认 20）  
- TypeScript 5.x strict  
- PostgreSQL 16  
- Redis 7  

> 说明：本地、Docker 基础镜像（如 `node:22-alpine`）、CI 均使用 Node 22，避免 20/22 混用导致依赖解析或原生模块差异。

## 5. 后端模块划分

按 NestJS Module 对齐领域（包路径示例 `apps/api/src/modules/*`）：

| 模块 | 职责 | MVP |
|------|------|:---:|
| `identity` | 用户、短信验证码、会话、资料 | ✅ |
| `workspace` | 个人空间创建与归属 | ✅ |
| `knowledge-base` | 知识库 CRUD、成员（P1） | ✅ / 成员 P1 |
| `tree` | TreeNode 树：创建/移动/排序/删除；**任意 parent 含 doc** | ✅ |
| `document` | 正文读写、`content_version`、冲突、修订快照 | ✅ |
| `share` | 分享 token 生成/校验/吊销 | ✅ |
| `audit` | 审计日志写入（覆盖冲突等 P0 关键写） | ✅ 最小 |
| `media` | 上传凭证与元数据 | P1 |
| `health` | `/health`、`/ready` | ✅ |

**依赖方向**：`document` → `tree` → `knowledge-base` → `workspace` → `identity`；`share` → `document` + `identity(可选访客)`。禁止循环依赖。

## 6. 核心领域机制

### 6.1 身份与短信登录

```
[客户端] 输入手机号 → POST /auth/sms/send
[API] 校验格式/限流 → 生成 6 位码 → Redis SET sms:{mobile} TTL 5min
     → SmsProvider.send（Mock 打日志）

[客户端] 提交手机号+验证码(+昵称) → POST /auth/sms/login
[API] 校验 Redis 码 → 不存在则创建 User(mobile_verified=true)+默认 Workspace
     → 创建 Session → Set-Cookie
```

| 规则 | 值（可配置） |
|------|----------------|
| 验证码长度 | 6 位数字 |
| TTL | 5 分钟 |
| 同一手机发送间隔 | ≥ 60 秒 |
| 同一手机日上限 | 例如 20 次 |
| 同一 IP 日上限 | 可配置 |
| 校验失败锁定 | 连续错误 N 次短时锁定 |

**会话**：

- Cookie 名：`sid`（示例）  
- 属性：`HttpOnly; Secure(prod); SameSite=Lax; Path=/`  
- Redis：`sess:{id}` → `{ userId, createdAt, ... }`，TTL 7～14 天  
- 登出：删 Redis + 清 Cookie  

**邮箱**：仅 `users.email` 可空字段，无验证流（P1 资料接口）。

### 6.2 文档树（folder / doc，doc 可挂子）

**模型**：

```
tree_nodes
  id, knowledge_base_id, parent_id NULLABLE,
  type: 'folder' | 'doc',
  title, sort_order, deleted_at, ...
```

**硬性约束（与 PRD Q8 一致）**：

1. `parent_id` 可为 `NULL`（库根）或 **同库任意未删除节点**（`folder` **或** `doc`）。  
2. 创建/移动时校验：  
   - parent 与 child 同 `knowledge_base_id`  
   - **不成环**（移动时从新 parent 向上走不能碰到自身）  
   - parent 未删除  
3. `type=doc` 在 `document_contents` 有 1:1 正文行；`folder` 禁止有正文。  
4. 删除：存在未删除子节点 → `409` / 业务错误，禁止删。  
5. 列表 API：按库返回树（邻接表一次查出，服务端或客户端建树）；节点量大时一期再做懒加载。

**API 形态（示意）**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/kbs/:kbId/tree` | 整树 |
| POST | `/kbs/:kbId/nodes` | body: `{ type, title, parentId? }` |
| PATCH | `/nodes/:id` | 重命名等 |
| POST | `/nodes/:id/move` | `{ parentId, sortOrder }` |
| DELETE | `/nodes/:id` | 软删，非空拒绝 |

### 6.3 正文保存与冲突（企业级乐观锁）

**表要点**：

```
document_contents
  node_id PK/FK,
  body_md TEXT,
  content_version INT NOT NULL,  -- 从 1 起
  updated_by, updated_at

content_revisions  -- 至少保留覆盖前快照
  id, node_id, version, body_md, created_by, created_at, reason
  -- reason: 'overwrite_on_conflict' | 'periodic' | ...
```

**保存算法**：

```
Save(docId, expectedVersion, body, userId):
  BEGIN
    row = SELECT ... FOR UPDATE WHERE node_id = docId
    if row.version != expectedVersion:
      ROLLBACK
      return 409 DOC_VERSION_CONFLICT { serverVersion, updatedBy, updatedAt, serverPreview? }
    if forceOverwrite flag:  // 仅冲突面板「覆盖」走专用接口或 force=true
      INSERT content_revisions(... old body, reason=overwrite_on_conflict)
      audit_log(...)
    UPDATE body, version = version+1, updated_by, updated_at
  COMMIT
  return { version, updatedAt }
```

**接口拆分建议**：

| 接口 | 用途 |
|------|------|
| `PUT /docs/:id/content` | 正常保存：必须 `expectedVersion` 匹配 |
| `POST /docs/:id/content/overwrite` | 冲突后强制覆盖：二次确认后调用；写 revision + audit |
| `POST /docs/:id/content/save-as` | 另存副本：新建 doc 节点+正文，可指定 parent |
| `GET /docs/:id/content` | 拉取最新（冲突选「加载最新」） |

**前端**：

- 防抖自动保存 2～5s；状态：`saved | saving | dirty | conflict`  
- `conflict` 时停止自动保存，展示三选一面板  
- 另存副本成功后可跳转新文档或留在当前  

### 6.4 分享链接

```
share_links
  id, token (high entropy), node_id (doc only),
  enabled, expires_at NULL, created_by, created_at
```

- 访客：`GET /share/:token` 校验 enabled/过期 → 返回只读标题+正文  
- 无需登录；限流防刷  
- 关闭分享：`enabled=false` 或删除记录  
- **不**随分享自动开放整棵子树（PRD）  

### 6.5 权限（MVP）

| 检查点 | 规则 |
|--------|------|
| 知识库/树/正文 | 当前用户为 Owner（创建者）或成员角色 ≥ 所需动作 |
| MVP 无成员时 | 仅 Owner 可读写；他人一律拒绝 |
| 分享 | 持有效 token 只读单 doc |
| 响应策略 | 无权限与不存在统一模糊为 404（防枚举，可配置） |

成员表（P1）：`kb_members(kb_id, user_id, role)`，`role ∈ owner|editor|reader`。

## 7. API 约定

### 7.1 风格

- REST + JSON  
- 前缀：`/api/v1`  
- 鉴权：Cookie Session；分享接口例外  
- 分页：`?page=&pageSize=`（列表类）  
- 时间：ISO-8601 UTC  

### 7.2 统一响应

成功：

```json
{
  "success": true,
  "data": {},
  "error": null,
  "requestId": "01J..."
}
```

失败：

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "DOC_VERSION_CONFLICT",
    "message": "文档已被他人更新",
    "details": {
      "serverVersion": 4,
      "updatedAt": "2026-07-12T10:00:00.000Z",
      "updatedBy": { "id": "...", "nickname": "..." }
    }
  },
  "requestId": "01J..."
}
```

### 7.3 关键错误码（节选）

| code | HTTP | 含义 |
|------|------|------|
| `UNAUTHORIZED` | 401 | 未登录 |
| `FORBIDDEN` / 模糊 `NOT_FOUND` | 403/404 | 无权限 |
| `NOT_FOUND` | 404 | 资源不存在 |
| `VALIDATION_ERROR` | 400 | 参数错误 |
| `SMS_RATE_LIMITED` | 429 | 短信过频 |
| `SMS_CODE_INVALID` | 400 | 验证码错误 |
| `DOC_VERSION_CONFLICT` | 409 | 正文版本冲突 |
| `NODE_HAS_CHILDREN` | 409 | 非空不可删 |
| `NODE_CYCLE_DETECTED` | 400 | 移动成环 |
| `SHARE_INVALID` | 404 | 分享无效 |

### 8.2 编辑器方案与 POC 标准

**MVP 主方案：CodeMirror 6 Markdown 源码编辑 + 分栏/切换预览。**

| POC 验收项 | 通过标准 |
|------------|----------|
| 语法 | 标题、列表、任务列表、代码块、表格、引用、链接、图片语法 |
| 体量 | 100KB MD 输入滚动与输入无明显卡顿（目标机型笔记本） |
| 保存 | 与 `expectedVersion` 联调；冲突面板三路径可测 |
| 安全 | 预览 HTML 经 DOMPurify；脚本不可执行 |
| 快捷键 | 常见加粗/标题等（可选但加分） |

**一期可选升级**：TipTap/ProseMirror 所见即所得，**存储仍建议保留 Markdown 或双写 schema 版本字段**，避免锁死二进制文档格式。

- **doc 节点与 folder 同样展示「新建子*」**

## 10. 安全设计

| 域 | 措施 |
|----|------|
| 传输 | 生产全站 HTTPS |
| 会话 | HttpOnly Cookie；CSRF：SameSite + 写操作 CSRF Token（若跨站场景需要） |
| 短信 | 频控、验证码单次失效、不返回「是否已注册」过于明确的枚举（产品可折中） |
| 注入 | ORM 参数化；禁止拼接 SQL |
| XSS | 服务端不信任 MD；客户端消毒；CSP 响应头 |
| 分享 | 高熵 token（≥128 bit 随机）；限流 |
| 上传 P1 | MIME/大小白名单；随机对象键 |
| 密钥 | 环境变量 / 密钥管理；不进 Git |
| 审计 | 登录失败、强制覆盖、分享开关、删除库等 |

**越权测试清单（提测必跑）**：他人库 ID 遍历、未分享 doc 直链、失效 token、只读角色写操作（P1）。

## 13. 关键时序

### 13.1 短信登录

```
User          Web           API           Redis        SMS
 |--mobile--->|             |              |            |
 |            |--send sms-->|              |            |
 |            |             |--set code--->|            |
 |            |             |--send------->|----------->|
 |            |<--ok--------|              |            |
 |--code----->|             |              |            |
 |            |--login----->|              |            |
 |            |             |--get code--->|            |
 |            |             |--create user/session----->|
 |            |<--Set-Cookie|              |            |
```

### 13.2 保存冲突

```
TabA/API: version=3
TabB: 保存成功 version=4
TabA: PUT expected=3
API: 409 CONFLICT
Web: 停自动保存 → 面板
  ├─ 加载最新 → GET content → version=4 继续编
  ├─ 覆盖 → POST overwrite → revision(3)+version=5
  └─ 另存 → POST save-as → 新 node
```

### 13.3 在 doc 下建子 doc

```
POST /kbs/{kb}/nodes
{ "type":"doc", "title":"子篇", "parentId":"{父docId}" }

API: 校验 parent 存在且同库 → 插入 tree_node
   → 若 type=doc 则插入 document_contents version=1
```

## 14. 风险与缓解

| 风险 | 等级 | 缓解 |
|------|------|------|
| 短信成本与可达 | 中 | Mock；频控；内网固定码开关仅 dev |
| 编辑器体验不达预期 | 高 | CodeMirror POC 门禁；预留 TipTap 一期 |
| 大树性能 | 中 | 索引 `(kb_id, parent_id, sort_order)`；虚拟化后置 |
| 冲突体验打扰 | 中 | 文案清晰；另存副本保底不丢字 |
| 会话 CSRF | 中 | SameSite；必要 crore 评估后加 CSRF token |
| 范围蔓延 | 高 | 严格按 PRD；变更走流程 |
