# 领域模型与数据库设计（Design-01）

> **版本**：v1.0  
> **状态**：对齐 [总体技术方案 v1.2](./00-总体技术方案.md) 与 [PRD v1.2](../prd/00-产品愿景与范围.md)  
> **数据库**：PostgreSQL 16  
> **ORM / 迁移**：Prisma Migrate  
> **ID 策略**：应用层生成 **ULID**（或 UUID v7）；全文以 `CHAR(26)` ULID 为例（可改为 UUID，全库统一即可）

## 1. 文档目标

将总体方案中的领域对象落实为：

1. 限界上下文与聚合边界  
2. 可迁移的表结构、约束、索引  
3. 软删除、版本、权限相关的数据完整性规则  
4. Redis 键设计（会话 / 短信 / 限流）  
5. 迁移与种子数据约定  

**不在本文**：完整 OpenAPI 字段（见 `docs/api/openapi.yaml`）；UI 状态。

## 2. 领域总览

### 2.1 限界上下文

| 上下文 | 聚合 / 主实体 | 说明 |
|--------|----------------|------|
| Identity | User | 账号、手机号、资料；Session 在 Redis |
| Workspace | Workspace | 个人空间 1:1 User；无对外 CRUD |
| Knowledge | KnowledgeBase, KbMember | 库与成员角色 |
| Tree | TreeNode | 邻接表树；folder / doc |
| Document | DocumentContent, ContentRevision | 正文与覆盖快照 |
| Share | ShareLink | 文档只读分享 |
| Audit | AuditLog | 追加写审计 |

`access` 为**领域服务**（无独立持久化表，读 User/Kb/Member/Share/Node）。

### 2.2 对象关系（概念）

```
User 1 ── 1 Workspace
Workspace 1 ── * KnowledgeBase
KnowledgeBase 1 ── * KbMember * ── 1 User
KnowledgeBase 1 ── * TreeNode
TreeNode (parent) 1 ── * TreeNode (children)   # parent 可为 folder 或 doc
TreeNode (type=doc) 1 ── 1 DocumentContent
DocumentContent 1 ── * ContentRevision
TreeNode (type=doc) 1 ── * ShareLink
User 1 ── * AuditLog (actor)
```

### 2.3 聚合边界与事务

| 聚合 | 事务边界示例 |
|------|----------------|
| User + Workspace | 注册登录：创建用户与默认空间同一事务 |
| KnowledgeBase | 创建库；软删库 |
| TreeNode | 创建节点（doc 时同事务插入 DocumentContent）；move；软删 |
| DocumentContent | 条件更新保存；overwrite（+ Revision + Audit） |
| ShareLink | 启用/禁用分享 |
| AuditLog | 通常与业务写同一事务提交 |

跨聚合：另存副本 = 新建 TreeNode+Content（可引用源节点只读），**不**改源 Content。

## 3. 标识与通用约定

### 3.1 主键

| 项 | 约定 |
|----|------|
| 类型 | `CHAR(26)` ULID（字典序近似时间有序） |
| 生成 | API 进程内生成，不依赖 DB sequence |
| 对外 | 知识库 `kbId`、节点 `nodeId` 等均为 ULID 字符串 |

### 3.2 时间戳

- `created_at` / `updated_at`：`TIMESTAMPTZ`，默认 `now()`  
- 应用层也可写，以 DB 默认兜底  
- API 对外 ISO-8601 UTC  

### 3.3 软删除

| 表 | 策略 |
|----|------|
| `tree_nodes` | `deleted_at` 可空；查询默认 `IS NULL` |
| `knowledge_bases` | 同上 |
| `users` | MVP 可不做注销；预留 `deleted_at` 可选 |
| `document_contents` | **不单独软删**；随 node 软删后不可访问 |
| `share_links` | 用 `enabled=false` 或硬状态；目标 node 软删后业务拒绝 |
| `audit_logs` / `content_revisions` | **不删**（追加型） |

### 3.4 枚举

PostgreSQL 使用 `TEXT` + `CHECK`，或 Prisma `enum` 映射。本文用 CHECK 语义描述。

## 5. 表结构明细

> 下列 `NOT NULL` / 默认值 / 索引为 **MVP 必须**；标注 P1 的列可延后迁移。

### 5.1 `users`

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| id | CHAR(26) | PK | ULID |
| mobile_e164 | VARCHAR(20) | NOT NULL, **UNIQUE** | E.164，如 `+8613800138000` |
| mobile_verified | BOOLEAN | NOT NULL DEFAULT true | 注册成功即为 true |
| nickname | VARCHAR(64) | NOT NULL | 可默认「用户xxxx」 |
| email | VARCHAR(255) | NULL | P1 资料；不做唯一强制 |
| password_hash | VARCHAR(255) | NULL | P1；argon2/bcrypt |
| avatar_url | VARCHAR(1024) | NULL | P1 |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |
| updated_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |
| deleted_at | TIMESTAMPTZ | NULL | 预留注销 |

**索引**：

- `UNIQUE (mobile_e164)`  
- 可选：`email` 在非空时唯一 → `UNIQUE` 部分索引 `WHERE email IS NOT NULL`（P1）

**业务规则**：

- 未验证手机号不得存在可登录用户（MVP 注册路径保证 verified=true）。

### 5.2 `workspaces`

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| id | CHAR(26) | PK | |
| owner_user_id | CHAR(26) | NOT NULL, **UNIQUE**, FK → users(id) | 个人空间 1:1 |
| name | VARCHAR(128) | NOT NULL | 默认「我的空间」 |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |
| updated_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |

**索引**：`UNIQUE (owner_user_id)`  

**规则**：注册事务内创建；无对外 API。

---

### 5.3 `knowledge_bases`

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| id | CHAR(26) | PK | 对外 kbId |
| workspace_id | CHAR(26) | NOT NULL, FK → workspaces(id) | |
| owner_user_id | CHAR(26) | NOT NULL, FK → users(id) | 冗余便于鉴权，与创建者一致 |
| name | VARCHAR(128) | NOT NULL | trim 后非空 |
| description | VARCHAR(2000) | NULL | |
| visibility | TEXT | NOT NULL DEFAULT 'private' | CHECK IN ('private','workspace')；MVP 主用 private |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |
| updated_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |
| deleted_at | TIMESTAMPTZ | NULL | 软删 |

**索引**：

- `(workspace_id) WHERE deleted_at IS NULL`  
- `(owner_user_id) WHERE deleted_at IS NULL`  
- `(updated_at DESC)` 可选，用于最近列表  

**规则**：

- 创建时 `owner_user_id` = 当前用户；可选同时插入 `kb_members(role=owner)`（P1 统一走成员表时推荐，见 §5.4）。  
- MVP 无成员表数据时：`access` 用 `owner_user_id` 判定 owner。

---

### 5.4 `kb_members`（P1 多成员；表结构 MVP 可先建）

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| id | CHAR(26) | PK | |
| knowledge_base_id | CHAR(26) | NOT NULL, FK → knowledge_bases(id) | |
| user_id | CHAR(26) | NOT NULL, FK → users(id) | |
| role | TEXT | NOT NULL | CHECK IN ('owner','editor','reader') |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |
| updated_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |

**约束**：

- `UNIQUE (knowledge_base_id, user_id)`  
- 同一库 **至多一个** `role='owner'`（可用部分唯一索引或应用保证；推荐应用 + 事务）

**索引**：

- `(user_id, knowledge_base_id)` — 我的库列表  
- `(knowledge_base_id, role)`  

**MVP 策略（冻结）**：

- **迁移即建表**，避免 P1 改 schema 痛苦。  
- MVP 写路径：创建 KB 时插入 owner 成员行 **推荐启用**（与 owner_user_id 双写，access 优先 members，回退 owner_user_id）。

---

### 5.5 `tree_nodes`

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| id | CHAR(26) | PK | 对外 **nodeId** |
| knowledge_base_id | CHAR(26) | NOT NULL, FK → knowledge_bases(id) | |
| parent_id | CHAR(26) | NULL, FK → tree_nodes(id) | NULL=库根；**可为 folder 或 doc** |
| type | TEXT | NOT NULL | CHECK IN ('folder','doc') |
| title | VARCHAR(512) | NOT NULL | 允许库内重名 |
| sort_order | INT | NOT NULL DEFAULT 0 | 同父下排序，升序 |
| created_by | CHAR(26) | NOT NULL, FK → users(id) | |
| updated_by | CHAR(26) | NULL, FK → users(id) | |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |
| updated_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |
| deleted_at | TIMESTAMPTZ | NULL | |

**索引**：

| 名 | 定义 | 用途 |
|----|------|------|
| idx_nodes_kb_parent_sort | `(knowledge_base_id, parent_id, sort_order) WHERE deleted_at IS NULL` | 拉子列表/建树 |
| idx_nodes_kb_title | `(knowledge_base_id, title) WHERE deleted_at IS NULL` | 标题搜索辅助 |
| idx_nodes_parent | `(parent_id) WHERE deleted_at IS NULL` | 子节点存在性（删除校验） |

**完整性（应用层强制，DB 难表达部分）**：

1. `parent_id` 若非空：parent 必须同 `knowledge_base_id`、`deleted_at IS NULL`。  
2. **禁止环**：move 时沿新 parent 上溯。  
3. 删除：不存在 `parent_id = id AND deleted_at IS NULL` 的子行。  
4. `type='folder'` 不得存在 `document_contents` 行。  
5. `type='doc'` 创建时 **同事务** 插入 `document_contents`。  
6. **不在 DB 用触发器强制 3–5 时，服务端单测必覆盖。**

**可选 DB 增强（非 MVP 必须）**：

- 触发器校验 parent 同库；  
- `EXCLUDE` 难做环检测，保持应用校验。

---

### 5.6 `document_contents`

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| node_id | CHAR(26) | **PK**, FK → tree_nodes(id) ON DELETE RESTRICT | 1:1 doc 节点 |
| body_md | TEXT | NOT NULL DEFAULT '' | Markdown；应用层 ≤ 2MB |
| content_version | INT | NOT NULL DEFAULT 1 | 单调递增，从 1 起 |
| updated_by | CHAR(26) | NULL, FK → users(id) | |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |
| updated_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |

**约束**：

- `CHECK (content_version >= 1)`  
- 应用层：仅当 `tree_nodes.type='doc'` 可插入  

**乐观锁（正常保存）**：

```sql
UPDATE document_contents
SET body_md = $1,
    content_version = content_version + 1,
    updated_by = $2,
    updated_at = now()
WHERE node_id = $3
  AND content_version = $4;
-- affected = 1 成功；0 则查版本决定 404 vs 409
```

**大小限制**：在 API 校验 `octet_length(body_md) <= 2097152`，不单靠 DB。

---

### 5.7 `content_revisions`

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| id | CHAR(26) | PK | |
| node_id | CHAR(26) | NOT NULL, FK → tree_nodes(id) | |
| version | INT | NOT NULL | 快照对应覆盖**前**的 content_version |
| body_md | TEXT | NOT NULL | 覆盖前正文 |
| reason | TEXT | NOT NULL | MVP: `overwrite_on_conflict` |
| created_by | CHAR(26) | NOT NULL, FK → users(id) | 执行覆盖的人 |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |

**索引**：

- `(node_id, version DESC)`  
- `(node_id, created_at DESC)`  

**约束**：

- `CHECK (reason IN ('overwrite_on_conflict'))` — MVP；后期扩展枚举需迁移  
- 可选 `UNIQUE (node_id, version, reason, created_at)` 不必强唯一（同一 version 理论上一次覆盖）  

**规则**：仅 overwrite 路径写入；**不做**每次保存 revision（防爆表）。

---

### 5.8 `share_links`

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| id | CHAR(26) | PK | |
| token | VARCHAR(64) | NOT NULL, **UNIQUE** | 高熵；存储原文（持有即权限）或 token_hash（见下） |
| node_id | CHAR(26) | NOT NULL, FK → tree_nodes(id) | 必须 type=doc |
| enabled | BOOLEAN | NOT NULL DEFAULT true | |
| expires_at | TIMESTAMPTZ | NULL | P1；NULL=永不过期（直到关闭） |
| created_by | CHAR(26) | NOT NULL, FK → users(id) | |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |
| updated_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |
| revoked_at | TIMESTAMPTZ | NULL | 可选，与 enabled 双写 |

**索引**：

- `UNIQUE (token)`  
- `(node_id) WHERE enabled = true`  

**Token 生成**：CSPRNG ≥ 128 bit，编码为 URL-safe base64 或 hex（长度计入 64 内）。  

**MVP 存储策略（冻结）**：

- **存 token 明文**于 DB（实现简单；依赖 DB 访问控制与备份加密）。  
- 一期可改为只存 `SHA-256(token)`，查询时哈希比对。

**规则**：

- 校验：`enabled` 且 (`expires_at` IS NULL OR > now()) 且 node 未删且 type=doc。  
- 关闭分享：`enabled=false`，写审计。  
- 同一 doc 允许多条历史 token；或 MVP 一文档一条有效（应用层：开启时禁用旧链）— **推荐 MVP：同一 node 仅一条 enabled=true**（部分唯一索引）：

```sql
CREATE UNIQUE INDEX uq_share_one_active
ON share_links (node_id)
WHERE enabled = true;
```

### 5.9 `audit_logs`

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| id | CHAR(26) | PK | |
| actor_user_id | CHAR(26) | NULL, FK → users(id) | 匿名分享访问可空 |
| action | VARCHAR(64) | NOT NULL | 如 `doc.overwrite_on_conflict` |
| resource_type | VARCHAR(32) | NOT NULL | `kb` / `node` / `share` / `auth` ... |
| resource_id | VARCHAR(64) | NULL | |
| metadata | JSONB | NOT NULL DEFAULT '{}' | 版本号、IP 哈希等 |
| request_id | VARCHAR(64) | NULL | 链路关联 |
| ip | VARCHAR(64) | NULL | 可选脱敏/截断 |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |

**索引**：

- `(created_at DESC)`  
- `(resource_type, resource_id, created_at DESC)`  
- `(actor_user_id, created_at DESC)`  
- `(action, created_at DESC)`  

**P0 action 枚举（应用常量）**：

| action | resource_type |
|--------|----------------|
| `doc.overwrite_on_conflict` | node |
| `share.enable` / `share.disable` | share |
| `node.delete` | node |
| `kb.delete` | kb |
| `auth.login_failed` | auth |
| `auth.locked` | auth |

只追加，不做更新删除。

## 6. 状态机

### 6.1 知识库

```
active (deleted_at IS NULL)
    │ delete
    ▼
deleted (deleted_at SET)
```

- 已删库：树与内容均不可通过业务 API 访问（联表过滤 kb.deleted_at）。  
- MVP 无恢复 API；数据保留可运维恢复。

### 6.2 树节点

```
active ──delete──► deleted
         ▲ 要求无 active 子节点
```

### 6.3 文档正文版本

```
version = n
  │ 条件更新成功
  ▼
version = n+1

version = n
  │ overwrite
  ▼
revision 保存 n 的 body → version = n+1
```

无「草稿/已发布」双状态字段（发布态为 P1）；MVP 仅 `body_md` 当前真源。

### 6.4 分享

```
enabled=true  ──disable──► enabled=false
     │
     └── expires_at 过期（P1）→ 业务视为无效
```

## 7. 访问控制的数据读路径

`access` 不落库，判定顺序建议：

### 7.1 读文档 `nodeId`

1. 若 actor = share token：  
   `share_links` 有效且 `node_id` 匹配 → **仅 read 该 node**  
2. 若 actor = user：  
   - 加载 node + kb（未删）  
   - 若 `kb_members` 有 role ∈ owner/editor/reader → 允许 read  
   - 否则若 `kb.owner_user_id = userId` → 允许 read  
   - 否则 deny → 404  

### 7.2 写文档 / 树

- member role ∈ owner/editor，或 owner_user_id 匹配。  
- reader / share → deny。

### 7.3 admin（成员、删库）

- role = owner 或 owner_user_id 匹配。

| 用途 | Key | Value | TTL |
|------|-----|-------|-----|
| Session | `sess:{sid}` | JSON `{ userId, createdAt, ... }` | 7d（滑动可刷新） |

| 短信发送间隔 | `sms:gap:{e164}` | 1 | 60s |
| 短信日计数 | `sms:day:{e164}:{yyyymmdd}` | count | 到次日 |
| 短信 IP 日计数 | `sms:ip:{ip}:{yyyymmdd}` | count | 到次日 |
| 登录失败 | `auth:fail:{e164}` | count | 15min |
| 登录锁定 | `auth:lock:{e164}` | 1 | 15min |
| 限流通用 | `rl:{scope}:{id}:{window}` | count | 窗口长 |

Session **不进 PostgreSQL**（MVP）。

## 9. 关键查询与写入 SQL 形态

### 9.1 拉取整树

```sql
SELECT id, parent_id, type, title, sort_order, updated_at
FROM tree_nodes
WHERE knowledge_base_id = $kbId
  AND deleted_at IS NULL
ORDER BY parent_id NULLS FIRST, sort_order, id;
```

应用层 O(n) 建树。

### 9.2 标题搜索

```sql
SELECT id, parent_id, type, title
FROM tree_nodes
WHERE knowledge_base_id = $kbId
  AND deleted_at IS NULL
  AND title ILIKE '%' || $q || '%'   -- 应用层转义 % _
ORDER BY updated_at DESC
LIMIT 50;
```

### 9.3 是否存在未删子节点

```sql
SELECT 1 FROM tree_nodes
WHERE parent_id = $nodeId AND deleted_at IS NULL
LIMIT 1;
```

### 9.4 创建 doc 节点（事务）

```sql
BEGIN;
INSERT INTO tree_nodes (... type='doc' ...);
INSERT INTO document_contents (node_id, body_md, content_version) VALUES ($id, '', 1);
COMMIT;
```

### 9.5 Overwrite（事务）

```sql
BEGIN;
-- SELECT content FOR 业务校验（可选）
INSERT INTO content_revisions (...);
UPDATE document_contents SET body_md=$1, content_version=content_version+1, ...
WHERE node_id=$id;
INSERT INTO audit_logs (...);
COMMIT;
```

## 10. Prisma 模型草稿（与迁移对齐）

> 实现时以 `schema.prisma` 为准；下列为契约级草稿。

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id             String    @id @db.Char(26)
  mobileE164     String    @unique @map("mobile_e164") @db.VarChar(20)
  mobileVerified Boolean   @default(true) @map("mobile_verified")
  nickname       String    @db.VarChar(64)
  email          String?   @db.VarChar(255)
  passwordHash   String?   @map("password_hash")
  avatarUrl      String?   @map("avatar_url") @db.VarChar(1024)
  createdAt      DateTime  @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt      DateTime  @updatedAt @map("updated_at") @db.Timestamptz(6)
  deletedAt      DateTime? @map("deleted_at") @db.Timestamptz(6)

  workspace      Workspace?
```

## 11. 迁移策略

| 项 | 约定 |
|----|------|
| 工具 | Prisma Migrate |
| 命名 | `YYYYMMDDHHMMSS_描述` |
| MVP 首迁 | 一次创建 §5 全部表（含 kb_members、audit） |
| 禁止 | 生产手工改表不留 migration |
| 数据回填 | 无历史数据 |
| 破坏性变更 | 需 expand/contract：先加列 → 双写 → 切读 → 删旧列 |

**种子（dev）**：

- 可选：固定测试用户需经短信 Mock 创建，不做弱密码种子强依赖。  
- e2e 用 API 注册流程。

## 12. 数据量与保留

| 数据 | 预估 / 策略 |
|------|-------------|
| tree_nodes | ≤ 5 万级整库；单库 ≤ 5k |
| document_contents.body_md | 单行 ≤ 2MB |
| content_revisions | 仅覆盖时增长；可后期按 node 保留最近 N 条 |
| audit_logs | 建议保留 ≥ 30 天；超期归档/删（运维任务，非 MVP 代码必做） |
| share_links | 轻量 |
