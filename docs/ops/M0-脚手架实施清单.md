# M0 脚手架实施清单（Monorepo 初始化）

> **阶段目标**：搭建 monorepo、Prisma、Docker Compose、基础 API/Web，并按 **Issue → 分支 → Commit → PR → CI → 合入** 落地。  
> **对照基线**：本仓库已合入状态（Design-01 v1.1、OpenAPI 0.1.1、Node 22）。  
> **适用读者**：需从零复现或在新环境重建工程基线的开发者。  
> **原则**：先文档与规范，再依赖与配置，再数据层，再 API，再 Web，最后 CI 与合入。

---

## 0. 前置条件

| 项 | 要求 |
|----|------|
| 运行时 | Node.js **22**（`node -v` 主版本为 22） |
| 包管理 | pnpm **9**（`corepack enable && corepack prepare pnpm@9.15.0 --activate`） |
| 容器 | Docker + Compose v2（本地起 PG/Redis） |
| 工具 | `git`、`gh`（远程 Issue/PR） |
| 上游文档 | 至少已有 `docs/design/01-领域模型与数据库.md`、`docs/api/openapi.yaml`（契约 SSOT） |

**阶段完成总标准**：

```bash
pnpm install
pnpm run ci          # Node 检查 + docs + prisma generate + typecheck + test + build
# 可选联调：
pnpm compose:up
# 根目录 .env 中 DATABASE_URL 与 Compose 一致后：
pnpm db:migrate      # 勿用裸 exec prisma（易 P1012 缺 DATABASE_URL）
pnpm dev:api         # GET http://localhost:3000/api/v1/health → success
pnpm dev:web         # http://localhost:5173 可打开并探测 health
```

## 1. 推荐创建顺序总览

按下列 **Phase** 严格顺序创建（后一阶段依赖前一阶段）。

| 顺序 | Phase | 产出 |
|:----:|-------|------|
| 1 | 仓库与协作规范（若空仓） | `.gitignore`、Node 锁定、Git 规范、CI 初版、Issue/PR 模板 |
| 2 | 产品/设计输入（若尚未有） | PRD、Design-00/01、OpenAPI（本清单默认已存在，仅引用） |
| 3 | 根目录 monorepo 配置 | `pnpm-workspace`、根 `package.json`、共享 tsconfig、`.env.example` |
| 4 | Docker Compose | PG 16 + Redis 7 |
| 5 | Prisma（置于 API 包内） | schema + init migration |
| 6 | API 工程 | Nest 入口、Prisma 模块、Health |
| 7 | Web 工程 | Vite + React 首页 |
| 8 | CI 增强与运维文档 | 全量 `pnpm run ci`、本地手册 |
| 9 | Issue / 分支 / PR 合入 | 远程落地 |

## 12. 推荐 Issue 拆分、分支与 PR 合入顺序

> 与本仓库真实历史兼容，并适合**从零重建**时的拆分粒度。

### 12.1 Issue 拆分（建议）

| Issue | 标题建议 | 交付物 | 验收 |
|-------|----------|--------|------|
| **I0** | chore: 工程基线与 Git 工作流 | 根规范、CI 初版、Issue/PR 模板、Node 22 | CI 可跑 check:docs |
| **I1** | docs: 流程/PRD/Design/OpenAPI（若未有） | 设计与契约 | 评审通过 |
| **I2** | chore: monorepo workspace 与 env 模板 | workspace、根脚本、tsconfig 包、`.env.example` | install 可解析 workspace |
| **I3** | chore: Compose PG+Redis | `deploy/compose/docker-compose.yml` | 容器 healthy |
| **I4** | chore: Prisma schema 与 init migration | schema + SQL（含部分唯一索引） | migrate deploy 成功 |
| **I5** | feat: API M0 Nest + Health + PrismaModule | apps/api 可 build/health | curl health |
| **I6** | feat: Web M0 Vite 骨架 | apps/web 可 build，代理 health | 首页可访问 |
| **I7** | ci: 全量 monorepo 门禁 + 本地运维文档 | ci.yml、ops 文档、README | 远程 Lint & Checks 绿 |

**合并策略（时间紧时可压缩）**：

- **最小 2PR**：`I0+I2+I3+I4+I5+I6+I7` 一个大 M0 PR（本仓库 #22 接近此形态）+ 安全清理小 PR。  
- **推荐 3～4PR**（评审友好）：  
  1. 规范与 docs 门禁  
  2. monorepo + compose + prisma  
  3. api + web + ci  

### 12.2 分支命名

```text
chore/init-engineering-baseline-<issue>
chore/m0-workspace-compose-prisma-<issue>
feat/m0-api-health-<issue>
feat/m0-web-shell-<issue>
ci/m0-pipeline-docs-<issue>
```

均从最新 `main` 拉出；**禁止**直推 main。

### 12.3 Commit 规范示例

```text
chore(repo): 初始化工程基线与 Git 工作流

Closes #N
```

```text
chore(repo): 落地 M0 monorepo 与 Prisma 基线

- pnpm workspace：api + web
- Prisma init 对齐 Design-01
- Compose PG/Redis；CI 全量门禁

Closes #N
```

### 12.4 PR 合入顺序（严格依赖）

```text
PR-A 规范/CI 初版
  └─ PR-B monorepo + Compose + Prisma
       └─ PR-C API Health
            └─ PR-D Web 骨架
                 └─ PR-E CI 终态 + ops 文档（可与 C/D 合并）
```

若单 PR 大爆炸合入 M0：合并后若 GitGuardian 报**示例伪凭证**，立刻跟进小 PR：

- 将 `.env.example` / CI 中账号改为 `USER:PASSWORD` 占位  
- Compose 使用明确本地口令（如 `localdevonly`）并写进 ops 文档  

### 12.5 合入门禁（每 PR）

1. PR 模板填写完整，`Closes #issue`  
2. **Lint & Checks** 通过  
3. 不 force-push main；合并后删除分支  
4. 不提交 `.env`、不提交真实密钥
