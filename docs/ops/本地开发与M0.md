# 本地开发与 M0 基线

## 前置

- Node.js **22**（`.nvmrc`）
- pnpm 9（`packageManager`）
- Docker（Compose：PostgreSQL 16 + Redis 7）

## 运行拓扑（宿主机 Docker / code-server + DinD）

本仓库常见两种落地形态。**在 code-server + DinD 环境中**，拓扑如下（与现状一致）：

```text
宿主机 Docker
├─ code-server
│  └─ IDE、终端、代码目录（工作区挂载，如 /workspace/.../yuque1）
│     · 这里执行的 pnpm / node / curl 127.0.0.1
│     · 默认不在内层 compose 网络里
│
└─ code-server-dind          ← 内层 Docker 引擎（docker.sock 指向这里）
   └─ 内层 Docker Compose（deploy/compose，网络如 compose_default）
      ├─ yuque1-dev          ← 可选：开发态前后端容器（与 PG/Redis 同网）
      │  ├─ API  (Nest watch, :3000)
      │  └─ Vite (HMR, :5173)
      ├─ yuque1-postgres     ← 服务名 postgres
      └─ yuque1-redis        ← 服务名 redis
```

### 这意味着什么

| 事实 | 含义 |
|------|------|
| Compose 在 **DinD 内** 起 | `-p 5432:5432` / `-p 3000:3000` 绑在 **DinD 主机**（如 `code-server-dind` / `172.22.0.2`），不是 code-server 自己的 `127.0.0.1` |
| code-server 终端里的 `localhost` | 是 **IDE 容器** 的 loopback；未做转发时 **碰不到** DinD 上发布的端口 |
| 服务名 `postgres` / `redis` | **仅**在内层 compose 网络内可解析；code-server 里直接跑的 Nest **进不了**该网络 |
| 同网容器互通 | `yuque1-dev` 内用 `DATABASE_URL=...@postgres:5432/...`、`REDIS_URL=redis://redis:6379` 即可 |

### 怎么选启动方式

| 场景 | 推荐做法 |
|------|----------|
| **真·本机 Docker**（笔记本 Docker Desktop 等，无 DinD） | `pnpm compose:up` 后，在宿主直接 `pnpm dev:api` + `pnpm dev:web`；`.env` 用 `localhost:5432/6379` |
| **code-server + DinD**（上图） | 1) `pnpm compose:up` 2) `pnpm db:migrate:docker` 3) 在 **compose 网络内**跑 API/Vite（容器 `yuque1-dev`）4) 若 IDE 要打开 `http://localhost:5173`，在 code-server 侧把 `localhost:3000/5173` **转发到 DinD 发布地址**（如 `172.22.0.2`） |
| 只迁库、宿主探不到 `:5432` | 一律 `pnpm db:migrate:docker`（脚本在 compose 网内对主机名 `postgres` 执行 migrate） |

> 判断是否 DinD 隔离：在 code-server 终端  
> `node -e "require('net').connect(5432,'127.0.0.1',()=>{console.log('ok');process.exit(0)}).on('error',e=>{console.error(e.message);process.exit(1)})"`  
> 若 `ECONNREFUSED` 而 `docker ps` 里 postgres 已 healthy → 即本拓扑，**不要**假设 `localhost` 映射可用。

### 与「把 code-server 接入 compose 网」的关系

- **理想**：`docker network connect compose_default <code-server 容器>`，让 IDE 内进程直连 `postgres`/`redis`。
- **现实（本环境）**：内层 DinD 的 `docker ps` **看不到** 外层 `code-server` 容器，无法从 DinD 侧 `network connect` 进去。
- **故默认策略**：依赖与（可选）前后端都放在 **内层 compose 网络**；IDE 只编辑挂载目录，浏览器经 **端口转发** 访问 Vite/API。

## 一键依赖

```bash
cp .env.example .env
# 确认仓库根目录 .env 中 DATABASE_URL 与 Compose 一致，默认：
# postgresql://yuque:localdevonly@localhost:5432/yuque1?schema=public
# （仅当「当前进程」能访问 DinD/宿主发布的 5432 时 localhost 才有效）

pnpm install
pnpm compose:up   # PG user/pass: yuque / localdevonly（仅本地）

# 推荐（自动加载根目录 .env，避免 P1012）：
pnpm db:migrate
# 若上一步连不上 localhost:5432（DinD）：
# pnpm db:migrate:docker

pnpm db:generate
```

### DATABASE_URL 常见错误

| 报错 | 原因 | 处理 |
|------|------|------|
| `P1012 Environment variable not found: DATABASE_URL` | 根目录无 `.env`，或用了裸 `exec prisma` 未加载 env | `cp .env.example .env` 后用 `pnpm db:migrate` |
| `P1001 Can't reach database server` / `ECONNREFUSED 127.0.0.1:5432` | Compose 未起，或 **DinD：code-server 的 localhost ≠ 端口发布点** | 1) `pnpm compose:up` 确认 healthy<br>2) 宿主探测 localhost:5432<br>3) 若拒绝：用 **`pnpm db:migrate:docker`**；跑 API 时进 compose 网（见上图 `yuque1-dev`） |
| `[redis] connect ECONNREFUSED 127.0.0.1:6379` | 同左：API 在 code-server 内起，Redis 在 DinD compose 网内 | 不要在隔离 shell 里直接 `pnpm dev:api`；改在与 redis 同网的容器内启动，或修好到 DinD 发布端口的转发 |

> 不要使用：`pnpm --filter @yuque1/api exec prisma migrate deploy`（不会读 monorepo 根 `.env`）。

### Nest 与 Prisma 的 env 一致性

- Prisma CLI：`apps/api/scripts/run-prisma.cjs` 加载根 `.env`
- Nest 运行时：`apps/api/src/load-env.ts` 在 `main.ts` 入口同样加载
- 请始终在**仓库根**维护 `.env`
- 容器内跑 API 时覆盖为服务名，例如：  
  `DATABASE_URL=postgresql://yuque:localdevonly@postgres:5432/yuque1?schema=public`  
  `REDIS_URL=redis://redis:6379`

## 启动

### A. 本机 Docker（进程能访问 localhost 映射端口）

```bash
# 终端 1
pnpm dev:api

# 终端 2
pnpm dev:web
```

- Web: http://localhost:5173 （代理 `/api` → API）
- API: http://localhost:3000/api/v1/health

### B. code-server + DinD（推荐与 PG/Redis 同网跑前后端）

示意（容器名 `yuque1-dev`，与文档拓扑一致）：

```bash
pnpm compose:up
pnpm db:migrate:docker

docker rm -f yuque1-dev 2>/dev/null || true
docker run -d --name yuque1-dev \
  --network compose_default \
  -p 3000:3000 -p 5173:5173 \
  -v "$PWD":/app -w /app \
  -e DATABASE_URL='postgresql://yuque:localdevonly@postgres:5432/yuque1?schema=public' \
  -e REDIS_URL='redis://redis:6379' \
  -e CORS_ORIGIN='http://localhost:5173' \
  -e COOKIE_SECURE=false \
  -e SMS_PROVIDER=mock \
  -e SMS_MOCK_CODE=123456 \
  -e API_PORT=3000 \
  -e SESSION_SECRET='local-dev-session-secret-at-least-32-chars-xx' \
  -e CHOKIDAR_USEPOLLING=true \
  node:22-bookworm \
  bash -lc 'corepack enable && corepack prepare pnpm@9.15.0 --activate && sleep infinity'

docker exec -d yuque1-dev bash -lc 'cd /app && pnpm dev:api > /tmp/api.log 2>&1'
docker exec -d yuque1-dev bash -lc 'cd /app && pnpm --filter @yuque1/web exec vite --host 0.0.0.0 --port 5173 > /tmp/web.log 2>&1'

# 健康检查（在 compose 网 / 容器内）
docker exec yuque1-dev curl -sS http://127.0.0.1:3000/api/v1/ready
```

若浏览器或 IDE 端口转发只认 **code-server 的 localhost**：把 `127.0.0.1:3000/5173` TCP 转到 DinD 发布地址（常见为 `code-server-dind` / `172.22.0.2` 的同端口）。日志：

```bash
docker exec yuque1-dev tail -f /tmp/api.log
docker exec yuque1-dev tail -f /tmp/web.log
```

## 一键冒烟（含 DinD 网络）

```bash
pnpm smoke
```

会依次：unit/typecheck/build → compose up → 自动选择 `db:migrate` 或 `db:migrate:docker` → 启动 API 验证 health/ready（postgres/redis true）。
在 DinD 下冒烟脚本会把 API 放进与 postgres 相同的 Docker 网络探测，逻辑与上文拓扑一致。
