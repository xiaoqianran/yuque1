## 1. 背景与问题

### 1.1 背景

团队与个人普遍存在「知识散落在聊天、网盘、本地 Markdown、多套 Wiki」的问题。语雀等产品证明了「结构化知识库 + 好用编辑器 + 可控分享」的价值，但存在：

- 数据与服务受第三方约束（可用性、合规、定制、成本）
- 内网 / 私有化场景下难以直接使用公有云产品
- 学习与工程上需要一套可掌控的完整实现作为基线

### 1.2 要解决的问题

| 痛点 | 用户表现 | 本产品应对 |
|------|----------|------------|
| 知识无归属 | 文档不知该放哪、难检索 | 空间 → 知识库 → 文档树的清晰层级 |
| 写作与发布割裂 | 本地写完再拷贝上传 | 在线编辑、草稿与发布一体 |
| 分享失控 | 链接满天飞、权限说不清 | 私有默认 + 显式公开/链接分享 |
| 协作门槛高 | 工具重、上手慢 | 先做好「个人写得爽、小团队读得懂」 |

### 1.3 非目标（本阶段明确不做的事）

- 不做「完整 1:1 像素级复刻语雀所有功能」
- 不做实时多人光标协同（OT/CRDT）作为 MVP
- 不做通用低代码 / 多维表 / 画板等泛生产力套件
- 不做面向不确定海量租户的商业化计费中台（可后期扩展）

## 2. 产品愿景

### 2.1 愿景陈述

> **让个人与小团队在可私有部署的环境中，用接近语雀的体验，沉淀、组织并安全分享结构化知识。**

### 2.2 产品定位

| 维度 | 定位 |
|------|------|
| **一句话** | 可自托管的知识库与文档协作工具（语雀核心能力子集） |
| **品类** | 知识库 / 文档 Wiki，而非 IM、网盘或项目管理 |
| **部署** | 优先支持私有化 / 自托管；架构预留多租户扩展 |
| **差异化重点** | 代码与数据可控、工程规范完整、核心路径体验扎实 |
| **对标** | 语雀（主对标）、Notion（信息架构参考）、Confluence（权限参考） |

### 2.3 产品原则

1. **结构优先于堆砌**：先把「库与树」做对，再堆插件式能力  
2. **私有默认、分享显式**：新建内容默认仅所有者可见  
3. **写作为第一生产力**：编辑、保存、预览路径必须短且稳  
4. **权限可解释**：用户能回答「谁能看到这篇文档」  
5. **小步可用**：每个里程碑结束都应有可演示的完整链路

## 3. 目标用户与场景

### 3.1 目标用户（Persona）

| Persona | 描述 | 核心诉求 |
|---------|------|----------|
| **个人创作者 P1** | 工程师 / 学生 / 写作者，独自整理笔记与长文 | 快写、好整理、可公开部分文章 |
| **小团队知识官 P2** | 5～30 人团队中维护规范与 Wiki 的人 | 分库管理、成员可读可写、链接分享给外围 |
| **只读消费者 P3** | 通过分享链接或受邀只读的人 | 打开快、目录清、移动端可读（基础） |
| **系统管理员 P4** | 负责部署与账号的人（二期加强） | 可部署、可备份、有基础审计 |

**一期主焦点**：**P1 + P2**。P3 通过「公开/链接」覆盖；P4 仅满足最小自托管运维。

### 3.2 核心使用场景

1. **个人知识沉淀**：注册 → 创建知识库「学习笔记」→ 写 Markdown 文档 → 树状整理  
2. **团队规范库**：创建团队空间知识库 → 邀请成员 → 共同维护接口规范 / 值班手册  
3. **对外分享**：将单篇文档生成分享链接，外发只读访问；可关闭链接  
4. **查找既有文档**：（二期增强全文检索）一期至少支持库内标题级浏览与基础筛选  

### 3.3 典型用户旅程（MVP）

```
访客/用户 ──手机号+验证码注册/登录──▶ 进入个人空间
                                          │
                                          ▼
                                   创建知识库（命名、简介、可见性）
                                          │
                                          ▼
                                   创建分组(folder) / 文档(doc) 树
                                          │
                                          ▼
                                   编辑文档（Markdown）→ 自动/手动保存 → 预览
                                          │
                                          ▼
                            （若版本冲突 → 企业级冲突处理，见 §7.1 / US-05）
                                          │
                                          ▼
                                   设置文档/库权限或生成分享链接
                                          │
                                          ▼
                                   他人打开链接只读查看
```

## 4. 产品范围分层

### 4.1 范围总览

| 层级 | 名称 | 目标 | 是否本 PRD 冻结 |
|------|------|------|-----------------|
| **MVP** | 可用的个人/小团队知识库 | 完成「写—管—享」闭环 | ✅ 是 |
| **一期增强** | 体验与检索补齐 | 接近日常可替换轻量 Wiki | 方向冻结，细节另开 PRD |
| **二期** | 协作与组织能力 | 评论、历史、组织管理等 | 仅列边界，不承诺排期 |
| **远期** | 平台化 | 开放 API、多维能力、商业化 | 非目标，仅占位 |

### 4.2 MVP 范围（In Scope）— 必须交付

#### A. 账号与身份

| 能力 | 说明 | 优先级 |
|------|------|:------:|
| **手机号 + 短信验证码注册/登录** | 手机号为唯一主标识；**必须完成短信验证**后方可注册成功或绑定 | P0 |
| 密码（可选增强） | 支持「验证码登录」；可选设置密码便于无短信场景（内网可接模拟短信网关） | P0 验证码 / P1 密码 |
| 邮箱（可选） | 个人资料中绑定，**不参与强制校验**；用于通知预留，MVP 不强制验证 | P1 |
| 登录态保持与退出 | Cookie/Session 或等价安全方案 | P0 |
| 个人资料（昵称、头像 URL、可选邮箱） | 最小字段；头像可二期本地上传 | P1 |
| 手机号变更 | 需旧号验证 + 新号验证；审计记录 | P1 |
| 密码/凭证重置 | 通过已验证手机号验证码重置 | P1 |

#### B. 空间与知识库

| 能力 | 说明 | 优先级 |
|------|------|:------:|
| 个人空间 | 用户注册后默认拥有 | P0 |
| 知识库 CRUD | 创建、重命名、归档/删除、简介 | P0 |
| 知识库列表与首页入口 | 最近访问/我的知识库 | P0 |
| 知识库可见性 | 私有 / 空间内成员可见（若暂无多成员则等价私有） | P0 |
| 多成员加入知识库 | 邀请（已注册手机号）、角色：管理员/编辑/只读 | P1 |

#### C. 文档与文档树

| 能力 | 说明 | 优先级 |
|------|------|:------:|
| **分组与文档分离** | 树节点类型：`folder`（分组/目录）与 `doc`（文档），见 §5.1 / §7.2 | P0 |
| **doc 下挂子节点** | **P0 硬性要求**：`doc` 可作为父节点，下挂 `folder` 和/或 `doc`（多级子树），对齐语雀「文档下子文档」 | P0 |
| 文档 CRUD | 创建、编辑、重命名、删除（软删优先） | P0 |
| 分组 CRUD | 创建、重命名、删除（删分组策略见业务规则） | P0 |
| 文档树 | 同一棵树、`parent_id` + `sort_order`；**任意节点均可作为 parent**（含 doc）；排序 MVP 可用上移下移 | P0 结构 / P1 拖拽 |
| 草稿与发布 | 至少支持「保存内容」；显式「发布」状态可选 | P0 保存 / P1 发布态 |
| 文档元信息 | 标题、更新时间、作者、**content_version** | P0 |
| 乐观锁保存 | 保存必须携带版本；冲突走企业级冲突方案（§7.1） | P0 |

#### D. 编辑与阅读

| 能力 | 说明 | 优先级 |
|------|------|:------:|
| Markdown 编辑 | 标题、列表、代码块、表格、引用、链接、图片 | P0 |
| 预览 / 阅读态 | 编辑预览或独立阅读页 | P0 |
| 自动保存 | 防抖保存 + 保存状态提示；**冲突后暂停自动保存直至用户决策** | P0 |
| 冲突处理 | 企业级乐观锁 + 三选一决策（§7.1），禁止静默覆盖 | P0 |
| 图片上传插入 | 上传至对象存储并插入文档 | P1 |
| 富文本所见即所得 | 与 MD 双模可后置，MVP 以 MD 为主 | P2 |

#### E. 权限与分享

| 能力 | 说明 | 优先级 |
|------|------|:------:|
| 默认私有 | 未授权不可访问 | P0 |
| 知识库成员角色 | 管理 / 编辑 / 只读（P1 多成员时具备） | P1 |
| 文档级分享链接 | 生成 token 链接、只读、可关闭、可设置过期（过期 P1） | P0 基础链接 / P1 过期 |
| 公开知识库或公开文档 | 互联网可读（无登录）— 需防刷与安全默认关 | P1 |
| 越权访问统一体验 | 404/无权限页，不泄露存在性（按安全策略） | P0 |

#### F. 平台与非功能（MVP 底线）

| 能力 | 说明 | 优先级 |
|------|------|:------:|
| 响应式基础 | 桌面优先；平板/手机可读（编辑体验桌面优先） | P0 阅读 / P1 移动编辑 |
| 基础审计 | 关键写操作日志（谁在何时改了什么资源） | P1 |
| 可部署性 | Docker Compose 一键本地/演示环境 | P0 |
| 可观测最小集 | 结构化日志 + 健康检查 | P0 |
| 备份说明 | 文档级说明 DB/OSS 备份方式 | P1 |

### 4.3 明确排除（Out of Scope）— MVP 不做

## 5. 信息架构

### 5.1 核心对象

```
User（用户：mobile 必填且已验证；email 可选）
  └── Workspace（空间，MVP：个人空间为主）
        └── KnowledgeBase（知识库）
              ├── Member（成员与角色，P1）
              └── TreeNode（统一树表，type 区分；parent 可为根 / folder / doc）
                    ├── type = folder  → 分组：仅标题/排序/子节点，无正文
                    │         └── 可再挂 folder | doc …
                    └── type = doc     → 文档：有正文，且【必须允许】下挂子节点
                              ├── DocumentContent（正文、content_version、updated_at…）
                              │     ├── ShareLink（分享链接，MVP 挂在该 doc）
                              │     └── ContentRevision（版本快照，MVP 可只保留覆盖前快照）
                              └── children: folder | doc …（多级子树，P0 必做）
```

**模型决策（Q3 冻结）**：采用 **「同树异构节点」**——`folder` 与 `doc` 分离类型、共用一棵文档树；**`doc` 与 `folder` 一样可以作为父节点挂载子树**（硬性要求，非可选优化）。  
不采用「页面即目录」（Confluence 纯页面树）作为 MVP 主模型，理由见 §7.2。

### 5.2 导航结构（MVP）

| 区域 | 内容 |
|------|------|
| 全局顶栏 | Logo、知识库切换/搜索入口（搜索可灰）、用户菜单 |
| 左侧 | 当前知识库文档树 |
| 主区 | 文档阅读 / 编辑 |
| 右侧（可选） | 大纲目录（P1）、文档信息 |

### 5.3 关键页面（MVP）

- 登录 / 注册  
- 知识库列表（工作台）  
- 知识库设置（名称、成员 P1、危险操作）  
- 文档阅读页  
- 文档编辑页  
- 分享落地页（只读）  
- 无权限 / 404  
- 账户设置（最小）

## 6. 角色与权限矩阵（MVP）

### 6.1 角色定义

| 角色 | 范围 | 说明 |
|------|------|------|
| 访客 Guest | 全局 | 未登录；仅可访问显式公开或有效分享链接 |
| 登录用户 User | 全局 | 可创建自己的知识库 |
| 知识库所有者 Owner | 单库 | 创建者；全部权限 |
| 管理员 Admin | 单库 | 管理成员与设置（不可转移所有权除非另定） |
| 编辑 Editor | 单库 | 读写文档树与内容 |
| 只读 Reader | 单库 | 只读 |

MVP 可简化实现：**Owner + Editor + Reader**，Admin 与 Owner 合并。

### 6.2 权限矩阵（知识库级）

| 动作 | Guest | Reader | Editor | Owner |
|------|:-----:|:------:|:------:|:-----:|
| 查看库内文档 | 仅公开/链接 | ✅ | ✅ | ✅ |
| 编辑文档 | ❌ | ❌ | ✅ | ✅ |
| 管理文档树 | ❌ | ❌ | ✅ | ✅ |
| 邀请成员 | ❌ | ❌ | ❌ | ✅ |
| 修改库设置/删除库 | ❌ | ❌ | ❌ | ✅ |
| 创建分享链接 | ❌ | ❌ | ✅ | ✅ |
| 关闭分享链接 | ❌ | ❌ | ✅ | ✅ |

> 文档级更细权限（单篇加密、单篇指定人）— **非 MVP**。

## 7. 用户故事与验收标准（MVP 核心）

> 格式：作为…我希望…以便…  
> 验收：Given / When / Then  

### US-01 注册登录（手机号强制验证）

**作为**新用户，**我希望**使用手机号并完成短信验证后注册/登录，**以便**安全开始使用知识库。

- Given 我输入合法手机号并获取验证码  
- When 验证码正确并提交注册（可同时设置昵称）  
- Then 账号创建成功（`mobile_verified=true`）并进入工作台（或引导创建首个知识库）  
- Given 验证码错误或过期  
- When 提交  
- Then 注册/登录失败，账号不处于「半注册可登录」状态  
- Given 未验证手机号  
- When 任何需登录接口  
- Then 拒绝访问  
- Given 我在资料中填写邮箱  
- When 保存  
- Then 邮箱可选保存；**MVP 不要求邮箱验证码**，不影响登录  

**补充**：登录主路径为「手机号 + 短信验证码」；若已设置密码，允许「手机号 + 密码」作为 P1 能力。

### US-02 创建知识库

**作为**登录用户，**我希望**创建知识库并命名，**以便**分类沉淀内容。

- Given 我在工作台  
- When 创建知识库「工程规范」  
- Then 列表出现该库，进入后为空文档树，默认私有  

### US-03 创建与编辑文档

**作为**编辑者，**我希望**在库中新建 Markdown 文档并保存，**以便**记录知识。

- Given 我在某知识库且具备编辑权  
- When 新建文档并输入 Markdown，触发保存  
- Then 刷新后内容仍在，更新时间变化，UI 显示「已保存」  

### US-04 文档树组织（分组 / 文档分离 + doc 子树）

**作为**编辑者，**我希望**先建分组再在分组下建文档，并在文档下继续挂子文档/子分组，**以便**按目录浏览与规划层级结构。

- Given 我在某知识库且具备编辑权  
- When 创建类型为 `folder` 的节点「接口规范」  
- Then 树中出现分组节点；点击分组仅展开/管理，不进入正文编辑器  
- When 在该分组下创建类型为 `doc` 的文档  
- Then 文档挂在该分组下，点击进入编辑/阅读  
- Given 某篇 `doc` 已存在  
- When 在其下创建子 `doc` 或子 `folder`  
- Then 左侧树在该文档节点下正确展示子节点；父子均可独立打开（doc 进编辑/阅读，folder 仅结构）  
- When 将某节点移动到另一 `doc` 下作为子节点  
- Then `parent_id` 更新成功，树无环，展示正确  
- Given 分组或文档下仍有子节点  
- When 删除该父节点  
- Then 按 §8 业务规则拦截（MVP 默认：有子节点则禁止删除，需先移出或删空）  

### US-05 自动保存与冲突（企业级方案）

**作为**编辑者，**我希望**并发编辑时系统按企业知识库惯例处理冲突，**以便**不丢字、不静默互相覆盖，且关键覆盖可追溯。

- Given 我打开文档时 `content_version=N`，本地编辑中  
- When 服务端版本已变为 `N+1`（他人、另一标签页或另一端已成功保存）  
- And 我触发自动保存或手动保存且仍携带 `expected_version=N`  
- Then 服务端返回明确冲突错误（如 `409 DOC_VERSION_CONFLICT`），**不写入正文**  
- And 前端 **立即暂停自动保存**，弹出冲突处理面板（见 §7.1）  
- And 用户在完成决策前不能静默继续覆盖  

**MVP 产品决策（Q2 冻结）**：采用 **乐观锁 + 冲突决策面板** 的企业 Wiki 方案（对齐 Confluence 等「保存时版本校验」模式，而非 Google Docs 实时合并）。详见 §7.1。

### US-06 分享链接只读

**作为**编辑者，**我希望**生成分享链接，**以便**让未加入知识库的人只读查看。

- Given 我对某文档开启分享  
- When 访客打开有效链接  
- Then 可只读查看内容，不能编辑  
- When 我关闭分享后访客再打开  
- Then 不可访问  

### US-07 私有不可见

**作为**所有者，**我希望**私有库内容默认不被他人访问，**以便**控制敏感信息。

- Given 文档/库为私有且无分享  
- When 其他登录用户或访客直接访问 URL  
- Then 无法读取正文  

### US-08 成员协作（P1）

**作为**所有者，**我希望**邀请成员并设为编辑/只读，**以便**小团队共维护。

- Given 我邀请用户 B 为 Editor  
- When B 登录进入该库  
- Then B 可编辑；Reader 仅可阅读

## 7.1 文档冲突处理（Q2 冻结 · 企业级方案）

### 行业借鉴结论

| 产品形态 | 冲突策略 | 是否适合本 MVP |
|----------|----------|----------------|
| Google Docs / 语雀实时协同 | OT/CRDT 实时合并 | ❌ 已排除实时协同 |
| Confluence / 传统企业 Wiki | **保存时乐观锁**；冲突则提示，用户选择放弃本地 / 覆盖 / 另存 | ✅ **主对齐** |
| Git 文档流 | 三路合并 | ⚠️ 体验重，编辑器内不做；可作远期 |
| 纯 LWW（最后写入获胜） | 静默覆盖 | ❌ 企业场景不可接受 |

**本产品采用**：**乐观并发控制（Optimistic Locking）+ 显式冲突决策 + 覆盖可审计**。  
对齐企业 Wiki「先保护数据，再让用户做主」的惯例，而不是实时合并。

### 机制要点

1. **版本字段**：每篇文档正文维护单调递增 `content_version`（整型即可）。  
2. **打开文档**：响应带上当前 `content_version` 与 `updated_at`、`updated_by`。  
3. **保存请求**：必须携带 `expected_version`；仅当与服务端一致时接受写入，成功后 `version+1`。  
4. **自动保存**：防抖（建议 2～5s）；一旦收到冲突，**停止自动保存**，直到用户处理完毕并重新同步版本。  
5. **禁止**：无版本号的写入、冲突后静默重试覆盖、多端 LWW。

### 冲突面板（用户三选一）

| 选项 | 行为 | 说明 |
|------|------|------|
| **A. 加载最新（放弃本地）** | 拉取服务端正文覆盖编辑器；本地未保存更改丢弃（可先提示） | 默认推荐项文案偏「安全」 |
| **B. 覆盖服务器（强制保存）** | 二次确认后，以当前本地内容写入；**服务端先将旧正文写入修订快照**再覆盖；`version+1`；记审计日志 | **MVP 必须支持**（企业常见能力，对应 Confluence overwrite） |
| **C. 另存为新文档** | 将本地内容保存为同库新 `doc`（标题默认「原标题 (冲突副本)」），原文档不改 | **MVP 必须支持**，避免用户两难丢字 |

### 审计与数据保留（企业底线）

- 选项 B 强制覆盖时：写 `audit_log`（who / when / doc_id / from_version / to_version）。  
- 覆盖前快照：MVP 至少保留 **被覆盖前一版** 正文（`content_revisions` 或等价）；完整历史时间线 UI 仍属一期增强。  
- 冲突事件可打点：`doc_conflict_shown` / `doc_conflict_resolve_{load|overwrite|save_as}`。

### 前端状态机（简图）

```
editing ──save──▶ saving ──ok──▶ editing (version++)
                    │
                    └──409──▶ conflict_blocked
                                │
                    ┌───────────┼───────────┐
                    ▼           ▼           ▼
                 load_latest  overwrite   save_as_copy
                    │           │           │
                    └───────────┴───────────┘
                                ▼
                         editing (已同步新 version)
```

### 与「多成员同时编辑」的产品口径

- MVP **不承诺**实时看见对方光标或段落锁。  
- 产品文案可提示：「若多人同时编辑同一篇，保存时可能需要处理冲突」。  
- 若未来做实时协同，冲突方案可演进，但不推翻 `content_version` 作为服务端权威版本的基线。

## 7.2 文档树模型（Q3 冻结 · 经验结论）

### 可选模型对比

| 模型 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 纯文档树（页面即节点） | 每页可有子页，无真正「空分组」 | 实现单一 | 难表达「先搭目录再补文」；空壳页污染 |
| B. **分组 / 文档分离（采用）** | `folder` + `doc` 同树异构 | 结构清晰；对齐语雀目录/分组心智；权限可沿树继承 | 多一个类型分支 |
| C. 独立目录表 + 文档表 | TOC 与内容完全分离 | 排版灵活 | 双写一致成本高，MVP 过重 |

### 冻结决策

- **采用模型 B**：`TreeNode.type ∈ {folder, doc}`。  
- **folder**：无 `DocumentContent`；可有子 `folder`/`doc`；支持重命名、排序、移动。  
- **doc**：  
  - 有且仅有一份正文（`DocumentContent`）；  
  - **【P0 硬性】允许且必须支持作为父节点**：其下可挂任意多个 `folder` / `doc` 子节点，支持多级嵌套；  
  - 子节点与父 `doc` 在树上是结构关系，**不共享正文**；编辑父文档不影响子文档内容。  
- **父节点规则（冻结）**：  
  - 知识库根、任意 `folder`、任意 `doc` **均可**作为 `parent`；  
  - **禁止**「仅 folder 可挂子、doc 为叶节点」的实现或临时降级；  
  - UI 在 `folder` 与 `doc` 节点上均提供「新建子分组」「新建子文档」入口（权限具备时）。  
- **删除**：父节点（`folder` 或 `doc`）下存在未删除子节点时 **禁止删除**（MVP）；需先移出或删空子树。`doc` 删除仍走软删除。  
- **分享**：MVP 仅针对 **单个 doc 正文** 生成分享链接；分享页默认不自动展开整棵子树权限（子树阅读权限随库权限/另开分享，二期可做「含子文档分享」）。  
- **移动**：变更 `parent_id` + `sort_order`；服务端必须校验 **禁止成环**（不能把节点移到自己的子孙下）。  
- **只有 `doc` 有正文**；`folder` 永不挂 `DocumentContent`。

### 为何要求 doc 下挂子节点

1. **对齐语雀心智**：长文/规范常见「一篇总览 + 多篇子文档」，用户会在文档节点下继续建子页。  
2. **避免假分组**：若 doc 只能做叶子，用户只能用空 folder 模拟，结构语义变差。  
3. **一次建模**：`parent_id` 通用指向任意 `TreeNode`，前后端树组件统一递归，避免后期改「叶约束」导致迁移。

### 为何仍保留 folder（不选纯页面树）

语雀类产品同时需要「纯目录分组」与「带正文的父文档」。分离类型可减少无内容的假页面，又满足 doc 子树；表结构一次到位。

## 8. 业务规则（摘要）

1. **删除**：知识库/文档/分组默认软删除；MVP 可不做回收站恢复 UI，但数据可恢复口径需在技术设计中保留。  
2. **父节点删除**：`folder` 或 `doc` 下存在未删除子节点时 **拒绝删除**，提示先清空或移出（MVP）。  
2.1 **doc 子树**：允许在 `doc` 下创建/移动 `folder`、`doc`；与在 `folder` 下操作规则一致；移动与创建均须校验同库、无环。  
3. **命名**：知识库名、节点标题去首尾空格；库内标题允许重名（用 ID 区分），UI 需可辨。  
4. **空库引导**：无节点时展示「创建分组 / 创建第一篇文档」空态。  
5. **账号**：手机号唯一；注册/登录必须验证码校验成功；邮箱可选且非身份主键。  
6. **短信**：验证码有效期、发送频率、日限额按安全基线（技术设计）；开发环境允许 Mock 网关。  
7. **分享 Token**：高熵、不可枚举；支持吊销。  
8. **内容安全**：渲染 Markdown 必须防 XSS；上传类型白名单。  
9. **限流**：验证码发送、登录失败、分享页访问需有基础限流。  
10. **冲突**：见 §7.1；任何成功覆盖必须可审计。

- MVP 不强制企业 SSO；**身份主路径为手机号 + 短信验证码**

### 11.3 产品风险

| 风险 | 影响 | 缓解 |
