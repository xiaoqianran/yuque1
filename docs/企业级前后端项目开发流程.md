# 复刻语雀 · 标准企业级前后端项目开发流程

> **文档定位**：本文档定义从 0 到 1 复刻「语雀（Yuque）」类知识库协作产品时，应遵循的**企业级前后端一体化研发流程**。  
> **适用范围**：产品、设计、前端、后端、测试、运维、项目管理及相关协作方。  
> **版本**：v1.0  
> **状态**：基线流程（可随项目演进修订）

---

## 1. 文档目标与原则

### 1.1 目标

在可控成本与质量下，按企业标准完成「语雀」核心能力复刻，包括但不限于：

- 空间 / 知识库 / 文档树
- 富文本与 Markdown 双模式编辑
- 协作与权限（组织、成员、角色、分享）
- 搜索、评论、历史版本
- 账号体系与基础运营能力

流程的核心产出不是「能跑的 Demo」，而是：**可发布、可观测、可回滚、可协作、可审计**的生产级系统。

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **业务驱动** | 需求以用户价值与业务指标为先，技术服务于交付 |
| **契约优先** | 前后端以 API / 事件契约为边界，先定接口再并行开发 |
| **质量内建** | 单测、联调、评审、CI 门禁贯穿全程，而非末尾补测 |
| **小步快跑** | 垂直切片（Vertical Slice）交付，避免大爆炸合并 |
| **可观测可回滚** | 上线默认具备日志、指标、链路与一键回滚能力 |
| **安全默认** | 鉴权、鉴权失败策略、敏感数据、XSS/CSRF 等默认纳入设计 |
| **文档即资产** | 需求、设计、接口、运维手册与代码同生命周期维护 |

## 2. 项目阶段总览

```
立项与调研 → 需求分析 → 架构与设计 → 项目初始化
    → 迭代开发（契约 → 实现 → 联调 → 提测）
    → 测试与验收 → 预发与发布 → 运维与复盘
```

| 阶段 | 主要角色 | 关键产出 | 出口标准（Definition of Done） |
|------|----------|----------|--------------------------------|
| 立项与调研 | 产品 / 技术负责人 | 立项书、竞品分析、范围边界 | 范围冻结、资源与里程碑确认 |
| 需求分析 | 产品 / 业务 | PRD、用户故事、验收标准 | 评审通过，优先级 P0/P1/P2 明确 |
| 架构与设计 | 架构 / 前后端 / 设计 | 技术方案、ER/时序、UI 规范、API 草案 | 设计评审通过，风险与依赖清单齐备 |
| 项目初始化 | 全栈 / 运维 | 仓库、环境、CI、规范落地 | 本地可跑通脚手架，CI 绿灯 |
| 迭代开发 | 研发 | 功能代码、单测、接口文档 | 通过 Code Review + CI |
| 测试与验收 | 测试 / 产品 | 测试报告、缺陷闭环 | P0 缺陷清零，验收通过 |
| 发布上线 | 运维 / 研发 | 发布单、回滚方案、监控大盘 | 灰度/全量策略执行完成 |
| 运维与复盘 | 全员 | 复盘纪要、改进项 | 改进项进入下迭代 backlog |

## 3. 组织与职责（RACI 摘要）

| 活动 | 产品 | 设计 | 前端 | 后端 | 测试 | 运维 | 技术负责人 |
|------|:----:|:----:|:----:|:----:|:----:|:----:|:----------:|
| 需求澄清 | A | C | C | C | C | I | C |
| 技术方案 | C | I | R | R | C | C | A |
| API 契约 | C | I | R | A | C | I | C |
| 功能实现 | I | I | R | R | I | I | C |
| 联调 | C | I | R | R | C | I | C |
| 测试用例 | C | I | C | C | A | I | I |
| 发布 | I | I | C | C | C | A | A |
| 线上事故 | I | I | R | R | C | A | A |

- **R** = Responsible（执行）  
- **A** = Accountable（最终负责）  
- **C** = Consulted（协商）  
- **I** = Informed（知会）

## 4. 阶段详解

### 4.1 立项与调研

**输入**：业务目标（例如：对内知识库 / 对外文档协作 SaaS）

**动作**：

1. 竞品拆解（语雀 / Notion / Confluence 等）能力矩阵  
2. 划定 **MVP / 一期 / 二期** 边界  
3. 评估合规要求（数据驻留、日志审计、敏感词等）  
4. 评估技术风险（协同编辑、大文档渲染、全文检索、权限模型）

**输出**：

- `docs/00-立项与范围.md`
- 里程碑（M0 脚手架、M1 MVP、M2 协作、M3 搜索与运营…）
- 粗估人天与依赖

**本项目建议 MVP 边界（示例）**：

- 账号登录注册
- 个人空间 + 知识库 + 文档 CRUD
- Markdown 编辑与预览
- 基础权限（私有 / 公开链接）
- 文档树与目录

非 MVP（可二期）：实时协同、评论 @、高级表格、开放 API、组织级管理后台等。

---

### 4.2 需求分析

**动作**：

1. 用户角色建模：游客、成员、管理员、空间所有者  
2. 用户故事 + 验收标准（Given / When / Then）  
3. 业务流程与异常路径（权限不足、文档冲突、分享过期）  
4. 非功能需求：性能、可用性、安全、可访问性  

**输出**：

- `docs/prd/` 分模块 PRD  
- 原型链接或 Figma  
- 需求优先级与迭代排期

**需求质量门禁**：

- [ ] 每个 P0 需求具备可测验收标准  
- [ ] 权限与数据归属规则写清  
- [ ] 空态 / 加载 / 错误态有产品说明  
- [ ] 埋点与核心指标（创建文档数、DAU、编辑时长等）已定义

### 4.3 架构与设计

#### 4.3.1 系统架构（建议形态）

```
                    ┌─────────────┐
                    │   CDN/OSS   │  静态资源 / 附件
                    └──────▲──────┘
                           │
┌──────────┐    ┌──────────┴──────────┐    ┌────────────────┐
│  Web 端  │───▶│  API Gateway / BFF  │───▶│  业务微服务/单体 │
│  (SPA)   │    └──────────┬──────────┘    └────────▲───────┘
└──────────┘               │                        │
                           ▼                        ▼
                    ┌─────────────┐         ┌──────────────┐
                    │  鉴权 / SSO │         │ DB / Cache /  │
                    └─────────────┘         │ Search / MQ   │
                                            └──────────────┘
```

初期可采用 **模块化单体（Modular Monolith）**，领域边界清晰后再拆服务，避免过早微服务化。

#### 4.3.2 领域划分（语雀向）

| 领域 | 职责 |
|------|------|
| Identity | 用户、登录、会话、OAuth |
| Workspace | 空间、组织、成员 |
| KnowledgeBase | 知识库、分组、文档树 |
| Document | 文档内容、版本、草稿、发布 |
| Permission | RBAC / 资源级授权 / 分享链接 |
| Collaboration | 评论、通知（可后置） |
| Search | 索引与检索 |
| Media | 图片 / 附件上传 |
| Audit | 操作日志 |

#### 4.3.3 设计产出清单

| 文档 | 负责人 | 说明 |
|------|--------|------|
| 总体技术方案 | 技术负责人 | 选型、边界、容量、风险 |
| 领域模型 / ER | 后端 | 核心表与状态机 |
| 时序图（登录、保存文档、分享） | 后端 | 关键路径 |
| API 契约（OpenAPI） | 后端主笔 / 前端会签 | 契约仓库或 monorepo 内统一维护 |
| UI 设计规范 / 组件库映射 | 设计 + 前端 | 颜色、字体、间距、组件态 |
| 安全设计 | 安全 / 后端 | 鉴权、加密、上传校验 |
| 部署与拓扑 | 运维 | 环境、依赖、扩缩容 |

**设计评审（ADR 可选）**：重大技术决策写入 `docs/adr/NNNN-title.md`。

### 4.4 技术选型建议（可按团队调整）

| 层级 | 推荐方向 | 备注 |
|------|----------|------|
| 前端框架 | React + TypeScript + Vite | 生态成熟，适合复杂编辑器 |
| 状态管理 | Zustand / Redux Toolkit | 按复杂度选型 |
| UI | Ant Design / 自研设计系统 | 企业后台友好 |
| 编辑器 | TipTap / ProseMirror / Milkdown | 语雀核心能力，需专项 POC |
| 后端 | NestJS / Spring Boot / Go (Gin/Kratos) | 统一语言栈优先 |
| API 风格 | REST + OpenAPI 3；必要时 GraphQL | 文档与协作以 REST 为主 |
| 数据库 | PostgreSQL | JSONB 适合文档元数据 |
| 缓存 | Redis | 会话、热点、限流 |
| 对象存储 | S3 兼容 / MinIO | 附件 |
| 搜索 | Elasticsearch / OpenSearch / Meilisearch | 按规模选型 |
| 消息 | Redis Stream / RabbitMQ / Kafka | 异步索引、通知 |
| 鉴权 | JWT + Refresh / Session + Cookie | Web 优先考虑 HttpOnly Cookie |
| 可观测 | OpenTelemetry + Prometheus + Grafana + Loki | 统一 trace_id |

**选型原则**：优先「团队熟练度 × 生态成熟度 × 运维成本」，核心编辑器必须做 POC 再定稿。

### 4.5 项目初始化（工程基线）

#### 4.5.1 仓库策略

推荐 **Monorepo**（pnpm workspace / Nx / Turborepo）：

```
yuque-clone/
├── apps/
│   ├── web/                 # 前端应用
│   └── api/                 # 后端应用
├── packages/
│   ├── api-client/          # 由 OpenAPI 生成的客户端
│   ├── shared-types/        # 共享类型 / DTO
│   ├── eslint-config/
│   └── tsconfig/
├── docs/
│   ├── prd/
│   ├── design/
│   ├── api/
│   └── ops/
├── deploy/
│   ├── docker/
│   └── k8s/ 或 compose/
├── scripts/
└── .github/workflows/ 或 .gitlab-ci.yml
```

也可前后端分仓，但 **API 契约必须有单一事实来源（SSOT）**。

#### 4.5.2 环境划分

| 环境 | 用途 | 数据 |
|------|------|------|
| local | 本地开发 | 个人 Docker Compose 数据 |
| dev | 联调集成 | 可重置测试数据 |
| test / staging | 提测与验收 | 近似生产结构 |
| prod | 生产 | 严格权限与备份 |

配置规范：

- 密钥不进仓库，使用 `.env.example` + 密钥管理（Vault / 云厂商 KMS）
- 同一套制品多环境部署，仅配置不同

#### 4.5.3 工程规范一次性落地

- [ ] EditorConfig / ESLint / Prettier / Stylelint  
- [ ] commitlint + Conventional Commits  
- [ ] husky + lint-staged（可选）  
- [ ] 统一目录与命名规范  
- [ ] README：快速启动 5 分钟跑通  
- [ ] Docker Compose：API + DB + Redis + MinIO  
- [ ] CI：lint + typecheck + unit test + build

## 5. 迭代开发流程（核心循环）

每个迭代（建议 1～2 周）按以下节奏运转。

### 5.1 迭代启动

1. 从 backlog 拉取已排优先级的用户故事  
2. 故事拆分为 **前后端任务** + **测试任务**  
3. 识别依赖（设计稿、第三方账号、搜索组件等）  
4. 召开迭代计划会，确认容量与风险  

### 5.2 契约优先开发（Contract-First）

```
PRD 确认 → OpenAPI / 事件契约评审 → Mock / 客户端生成
         → 前后端并行实现 → 契约测试 → 真联调
```

**规则**：

1. 接口变更必须先改契约并通知消费方  
2. Breaking Change 走版本策略（URL 版本或兼容字段）  
3. 错误码与错误体结构全局统一  

**统一响应示例**：

```json
{
  "success": true,
  "data": {},
  "error": null,
  "requestId": "01H..."
}
```

错误时：

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "DOC_NOT_FOUND",
    "message": "文档不存在或无权访问",
    "details": {}
  },
  "requestId": "01H..."
}
```

### 5.3 分支与 Git 工作流

推荐 **Trunk-Based + 短生命周期特性分支** 或 **GitHub Flow**：

| 分支 | 说明 |
|------|------|
| `main` | 受保护，始终可发布 |
| `feat/*` | 功能开发 |
| `fix/*` | 缺陷修复 |
| `chore/*` | 工程与依赖 |
| `release/*` | 可选，用于发布窗口冻结 |

**提交规范（Conventional Commits）**：

```
feat(doc): 支持文档草稿自动保存
fix(auth): 修复 refresh token 过期未跳转登录
docs(api): 补充知识库成员接口说明
refactor(editor): 抽取 slugs 工具函数
test(kb): 补充知识库权限单测
```

**合并要求**：

1. PR 模板填写：背景、方案、测试证据、风险、回滚  
2. 至少 1 名 Reviewer Approve（核心模块 2 人）  
3. CI 全绿  
4. 禁止 force-push 到 `main`  
5. 大 PR（>400 行有效改动）应拆分  

### 5.4 前端开发规范要点

- 页面 = 路由 + 容器 + 展示组件；业务逻辑下沉 hooks / services  
- 服务端状态用 React Query / SWR，本地 UI 状态就近管理  
- 路由级 code-splitting；编辑器等重模块懒加载  
- 无障碍：焦点、键盘、对比度纳入基础清单  
- 关键文与权限：接口 403 与 UI 隐藏双重处理  
- 关键态统一：loading / empty / error / skeleton  

**编辑器专项**：

- 内容格式 schema 版本化（便于迁移）  
- 自动保存：防抖 + 冲突检测（updatedAt / version）  
- 大文档性能：虚拟滚动、按块渲染 POC 验收  

### 5.5 后端开发规范要点

- 分层清晰：Controller / Application / Domain / Infrastructure  
- 事务边界在应用服务层；禁止 Controller 堆业务  
- 权限校验靠近资源访问点（避免「查到再判断」导致信息泄露）  
- 幂等：创建类写接口支持 `Idempotency-Key`（按需）  
- 分页、排序、过滤参数统一  
- 软删除与审计字段（created_at / updated_at / deleted_at / operator）  
- 数据库迁移工具（Flyway / Prisma Migrate / golang-migrate）强制  

### 5.6 联调

1. 使用 dev 环境真实服务，关闭仅 Mock 路径  
2. 对照验收标准逐条打勾  
3. 记录接口不一致项，当天关闭或建缺陷  
4. 关键路径与网络弱网（Chrome throttling）至少覆盖核心写操作  

### 5.7 代码评审清单（精简版）

**通用**：

- [ ] 需求是否完整覆盖，有无过度设计  
- [ ] 命名与边界清晰，无神秘常量  
- [ ] 错误处理与日志是否足够  
- [ ] 无密钥 / 个人信息硬编码  

**前端**：

- [ ] 无危险 `dangerouslySetInnerHTML` 未消毒用法  
- [ ] 列表 key 稳定；无多余 re-render 热点  
- [ ] 权限与空态处理完整  

**后端**：

- [ ] SQL / ORM 无 N+1 与注入风险  
- [ ] 鉴权与越权测试是否考虑  
- [ ] 迁移可回滚或具备前向兼容

## 6. 质量保障体系

### 6.1 测试金字塔

| 层级 | 责任 | 覆盖重点 |
|------|------|----------|
| 单元测试 | 研发 | 领域逻辑、工具函数、纯组件 |
| 契约 / 接口测试 | 后端 + 前端 | OpenAPI 一致性、主要 API |
| 集成测试 | 后端 | DB、缓存、权限组合 |
| E2E | 测试 / 前端 | 登录 → 建库 → 写文档 → 分享 |
| 探索性 / 专项 | 测试 | 安全、性能、兼容性 |

### 6.2 质量门禁（CI）

合并到 `main` 前必须：

1. Lint + TypeCheck 通过  
2. 单测通过，核心模块覆盖率不低于团队约定阈值（建议增量 ≥ 70%）  
3. 构建成功  
4. 关键安全扫描（依赖漏洞 SAST 基线）  
5. （可选）预览环境自动部署  

### 6.3 缺陷管理

| 级别 | 定义 | 处理时限（工作日，可按 SLA 调整） |
|------|------|----------------------------------|
| P0 | 核心链路不可用 / 数据损坏 / 严重安全 | 立即响应，阻断发布 |
| P1 | 主功能受损有临时方案 | 24～48h |
| P2 | 次要功能问题 | 当前或下迭代 |
| P3 | 体验优化 | backlog |

缺陷必须可复现：环境、账号、步骤、期望、实际、日志 / requestId。

### 6.4 非功能测试基线（示例）

| 场景 | 指标建议 |
|------|----------|
| 文档保存 API | P99 < 300ms（不含大附件） |
| 文档打开（已缓存） | 首屏有效内容 < 2s |
| 知识库目录 1k 节点 | 展开/滚动可用 |
| 同时在线（目标容量） | 压测报告备案 |
| 附件上传 | 类型白名单 + 大小限制 + 病毒扫描（按需） |

## 7. 发布与运维

### 7.1 发布流程

```
代码冻结 → 变更评审 → 构建制品 → 部署 staging
  → 冒烟 / 回归 → 发布审批 → 灰度 prod → 观察 → 全量
```

**发布单必备**：

- 变更内容与影响面  
- 数据库迁移说明（是否锁表、预计耗时）  
- 配置变更  
- 回滚方案（应用回滚 + 数据迁移回滚策略）  
- 监控观察项与负责人  
- 沟通话术（对内/对外）  

### 7.2 灰度与回滚

- 前端：静态资源版本化，支持按比例或按用户灰度  
- 后端：多副本滚动更新；不兼容变更使用双写 / 特性开关  
- 回滚目标：应用 **15 分钟内** 可回退到上一稳定版本  

### 7.3 可观测与告警

最低标准：

- **日志**：结构化 JSON，含 `requestId` / `userId`（脱敏）  
- **指标**：QPS、错误率、延迟、DB 连接池、队列堆积  
- **链路**：关键写路径全链路 trace  
- **告警**：5xx 突增、延迟飙升、磁盘 / 证书过期、登录失败率异常  

### 7.4 备份与容灾

- 数据库：定时全量 + 增量 / WAL；定期恢复演练  
- 对象存储：跨区域复制（按等级）  
- 恢复演练至少每季度一次  

### 7.5 安全运维基线

- HTTPS 全站；安全响应头（CSP、HSTS 等）  
- 密码哈希（bcrypt / argon2）；登录限流与锁定策略  
- 分享链接 token 高熵、可吊销、可过期  
- 上传 MIME 校验 + 随机存储名  
- 管理操作审计日志保留策略明确  
- 依赖与镜像定期漏洞修复

## 9. 里程碑建议（复刻语雀）

| 里程碑 | 目标 | 主要交付 |
|--------|------|----------|
| **M0 工程基线** | 可协作开发 | 仓库、CI、规范、本地一键启动 |
| **M1 账号与空间** | 能登录建空间 | 注册登录、个人空间、成员基础 |
| **M2 知识库与文档** | 能写能读 | 知识库、文档树、Markdown CRUD、草稿 |
| **M3 权限与分享** | 能控可见性 | RBAC、公开/链接分享、基础审计 |
| **M4 编辑体验** | 接近可用 | 富文本能力增强、目录、图片上传 |
| **M5 搜索与稳定** | 可内测 | 全文检索、性能优化、监控完备 |
| **M6 协作增强** | 可扩大使用 | 评论、通知、历史版本 diff |
| **M7 生产加固** | 可正式上线 | 安全加固、备份演练、SLA、运营后台 |

每个里程碑结束必须有：**演示 + 测试报告 + 已知问题列表 + Go/No-Go 决策**。

## 11. 风险与应对（语雀类产品特有）

| 风险 | 影响 | 应对 |
|------|------|------|
| 编辑器选型不当 | 返工巨大 | M0/M1 强制 POC，锁定 schema |
| 权限模型复杂 | 越权漏洞 | 统一鉴权中间件 + 权限矩阵测试 |
| 大文档性能 | 体验崩坏 | 分块存储/渲染、性能预算 |
| 协同冲突 | 数据覆盖 | 版本号 CAS；实时协同后置 OT/CRDT |
| 搜索延迟/不准 | 可用性差 | 异步索引 + 最终一致性说明 |
| 范围蔓延 | 延期 | MVP 冻结，变更走变更委员会 |
| 过早微服务 | 交付慢 | 模块化单体优先 |
